# API

This document records the public interface surface. Runtime internals are exposed through structured result metadata rather than separate endpoint families.

## Agent

| Method | Endpoint | Purpose |
|---|---|---|
| POST | `/api/agent/message` | Send one message and receive an agent result. |
| WS | `/ws/agent` | Stream agent turn events and final result. |
| GET | `/api/agent/approvals/pending` | List pending approvals. |
| POST | `/api/agent/approvals/:id/resolve` | Resolve an approval request. |

## Knowledge and memory

Knowledge and memory APIs remain storage-facing. Runtime memory write planning is exposed through turn metadata as `memory_write_plan`; it is not a persistence endpoint.

## Runtime metadata in agent result

The current architecture expects these structured metadata fields when available:

- runtime_state_snapshot
- task_signal
- action_trace
- artifact_records
- output_summary
- final_response
- memory_write_plan
- turn_trace
- truth_report
- stability_report

## Files and artifacts

Artifacts produced by the runtime are represented by `artifact_records`. The record is the stable handoff object for frontend display and later file operations.

## Runtime health

Health and version endpoints should read from the truth and stability reports where possible. Documentation must not duplicate runtime facts that can be generated by the source code.
