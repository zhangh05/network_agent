# v3.10 Release Notes

## Summary

v3.10 is a 12-phase architecture upgrade from a "local network agent toolbox" to a
**durable, auditable, manifest-driven Agent Runtime + Task Workbench**.

Every tool execution path has been unified through ToolRuntimeClient. Every runtime state
is checkpointed and recoverable. Every approval flows through interrupt/resume primitives.
Every workspace/session/run/job boundary is enforced with zero default fallbacks.

## Breaking Changes

| Area | Before (v3.8/3.9) | After (v3.10) |
|------|------------------|---------------|
| Approval resolve body | `{allowed: true/false}` | `{decision: "approve"}` — old body returns 400 |
| Workspace ID | Defaulted to "default" everywhere | Explicit required on mutating endpoints |
| AgentSession | `workspace_id = "default"` | `workspace_id = ""` |
| submit_user_message | `workspace_id="default"` | `workspace_id` required (no default) |
| ToolResult | Some `redacted=False` | All `redacted=True` by default |
| retry_step | Hardcoded pattern matching | Reads from manifest idempotency |
| RiskPolicy | Own classification engine | Reads from Capability Manifest |
| Canonical registry | Had dispatch bypass path | All execution via ToolRuntimeClient only |

## Removed Legacy Paths

- `_tool_approvals` / dual approval store — single ApprovalStore only
- `canonical_registry.dispatch()` — deleted
- `AgentInspector` component — deleted (624+319 lines)
- `allowed` body compatibility in resolve endpoint
- `or "default"` fallbacks across 15 files (op.py, facade.py, hook_runner, command_system, memory_write, etc.)
- Hardcoded idempotency patterns in `control.py` (`_NON_IDEMPOTENT_PATTERNS, _is_destructive, _is_idempotent`)
- Inspector panel CSS (~94 lines)
- Sticky fade pseudo-element on chat container

## Known Limitations

- `graph_runner.py` (LangGraph mode) not fully wired to SSE event bus
- Frontend does not consume new runtime APIs yet (by design — phased UI rollout)
- Module-level tools (cmdb, pcap, config_*) retain internal `"default"` parameter defaults behind validated routes
- Eval script requires actual trajectory data to produce meaningful output
- Full agent-run regression tests (31) require live LLM infrastructure

## Architecture Principles

1. **Single runtime** — all tool execution via ToolRuntimeClient
2. **Single approval** — one ApprovalStore, interrupt/resume primitives
3. **Manifest-driven** — policy/approval/retry derived from CapabilityManifest
4. **Durable state** — checkpointable, resumable, cancellable at any point
5. **Auditable delivery** — trajectory + eval + audit report for every task
6. **Zero default** — no implicit workspace, session, or runtime fallbacks
7. **Redacted by default** — all tool outputs, events, checkpoints redacted
